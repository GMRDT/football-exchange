pre-set variables):

## What to observe (F3.5 acceptance)

Run in the Supabase SQL editor (prod) during the match:

1. **Event appears ≤60 s after it happens on TV** (for a player in our 213):
   ```sql
   select me.minute, et.code, p.full_name, me.api_event_key, me.created_at
   from match_events me
   join players p on p.id = me.player_id
   join event_types et on et.id = me.event_type_id
   join matches m on m.id = me.match_id
   where m.api_fixture_id in (1539000, 1489370)
   order by me.created_at desc;
   ```
2. **Fair value jumps once per event** (goal +8%, assist +5%, etc. — MARKET_ENGINE.md §1.4)
   and **price drips over ~3 min**, then the delta row disappears:
   ```sql
   select p.full_name, d.total_pct, d.applied_pct, d.created_at
   from pending_price_deltas d join players p on p.id = d.player_id;
   ```
3. **No double-pricing**: `api_event_key` is UNIQUE; a brace in the same minute must show
   two rows, the second suffixed `#2` — never a missing or duplicated price move.
4. **Price history converges P → V** after the drip (reason `tick` rows).
5. **Invariants clean before & after**: `prod-env pnpm check-invariants`.
6. **No silent cron failures** (pg_net swallows function errors — check responses):
   ```sql
   select status_code, count(*) from net._http_response
   where created >= now() - interval '2 hours' group by 1;
   ```
   Anything ≥400 here means a function is failing while cron reports "succeeded".
7. **API budget** (Pro: 7,500/day): live polling costs ~1 req/min per live fixture.
